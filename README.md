# Family Asset Playbook (FAPM)

> A reusable family asset planning playbook for lives that keep changing.

## What This Is

Family Asset Playbook is an advisor-facing workflow for high-level household asset planning.
An advisor collects client information through a structured questionnaire, and the system generates a playbook that clients can read, discuss, and revisit later.

Live demo:

- [https://rhozero-div-family-asset-playbook.hf.space/](https://rhozero-div-family-asset-playbook.hf.space/)

This project is **not investment advice** and **not a product recommendation tool**.
Its current role is closer to:

- Advisor side: intake, validation, projection, and playbook generation
- Client side: reading the plan, understanding timing pressure, and discussing adjustments

## Current Product Flow

The actual production chain in this repo is:

1. An advisor fills out the web questionnaire
2. The frontend serializes the intake into YAML
3. The engine loads the YAML, projects future cash flows, and separates funds into mental buckets
4. The system renders a client-readable Markdown / HTML playbook

If documentation and code ever disagree, treat the **actual YAML produced by the current web questionnaire** and the **actual output generated by the current engine** as the source of truth.

## Public Demo Structure

This repo includes a dual-demo deployment setup:

- `frontend/`
  - A standalone Next.js demo frontend
  - Uses static export with `output: "export"`
  - Intended for Cloudflare Pages
- `backend/`
  - Docker entrypoint for Hugging Face Spaces
  - Reuses the current `web.app:app`
  - Intended for the real questionnaire / playbook flow

Recommended split:

- Cloudflare Pages: public-facing mock or marketing demo
- Hugging Face Spaces: the actual FastAPI questionnaire and playbook generator

## Current Calculation Rules

- Cash flow is modeled on a yearly basis
- Major events are settled at the **end of the event year**
- Charts default to the `measurement_end_year`
  - If the questionnaire already contains major events, the default is the last event year
  - Otherwise it falls back to `current_year + 30`
- If a savings insurance policy has a `linked_account`, it is merged into that bucket's initial allocation
- Any savings insurance amount without a valid `linked_account` goes into the surplus account
- Portfolio-level percentile bands are calculated from the **full portfolio path**, not by summing bucket-level percentiles

## Current Questionnaire Structure

The web questionnaire currently follows this structure:

1. Family members and life-milestone context
2. Income
   - Per member
   - Includes future income start age for minors / not-yet-working members
   - Includes post-retirement income estimates
3. Regular spending
   - Per member
   - Includes retirement spending coefficients
   - Includes household-level debt and extra spending
4. Major spending plan
5. Current assets
   - Real estate
   - Financial assets
   - Liquidity target
   - Savings insurance
6. Current insurance
7. Risk preference and projection assumptions
8. Advanced projection assumptions

## Current Playbook Reading Order

The generated playbook is currently organized in this order:

1. Executive Summary
2. Client Overview
3. Asset Projection
4. Allocation Execution Plan

## External Dependency

This project depends on a separately maintained QMC repository:

- [`rhozero-div/Quasi-Monte-Carlo-Generator`](https://github.com/rhozero-div/Quasi-Monte-Carlo-Generator)

The Monte Carlo engine imports it directly with `from qmc import ...`.
Cloning this repo alone is therefore not enough if you want the full projection and charting flow.
Your Python environment also needs to be able to resolve `qmc`.

Recommended approach:

1. Activate this project's own virtual environment `.venv`
2. Install `Quasi-Monte-Carlo-Generator` into the same environment, or add that repo root to `PYTHONPATH`

If `qmc` is unavailable, playbook generation may fail with errors such as `No module named 'qmc'`, or degrade to a version without return fan charts depending on the execution path.

### Recommended Local Setup

```bash
cd /path/to/family-asset-playbook

source .venv/bin/activate

pip install -r web/requirements.txt

pip install -e /path/to/Quasi-Monte-Carlo-Generator
```

If you do not want an editable install, you can also set:

```bash
export PYTHONPATH="/path/to/Quasi-Monte-Carlo-Generator:$PYTHONPATH"
```

## Local Run

For local use, enable server-side persistence by default:

```bash
FAPM_ENABLE_SERVER_STORAGE=1 .venv/bin/python -m uvicorn web.app:app --host 127.0.0.1 --port 8000
```

With `FAPM_ENABLE_SERVER_STORAGE=1`:

- client YAML files and indexes are written into `profiles/`
- saved questionnaires can be reopened locally
- local testing behaves like the advisor workflow

Without it:

- the app runs in public-demo mode
- anonymous submissions are not saved on the server
- saved-client browsing is disabled

Use non-persistent mode only for public demos, anonymous trials, or explicit no-storage scenarios.

### Cloudflare Pages

```bash
cd frontend
npm install

```

Recommended Pages settings:

- Build command: `npm run build`
- Output directory: `out`

### Hugging Face Spaces

The current deployment can use:

- [Dockerfile](Dockerfile)
- [backend/Dockerfile](backend/Dockerfile)
- [backend/requirements.txt](backend/requirements.txt)
- [.github/workflows/sync-to-huggingface.yml](.github/workflows/sync-to-huggingface.yml)

The Docker entrypoint runs:

```bash
uvicorn web.app:app --host 0.0.0.0 --port 7860
```

Notes:

- The Hugging Face deployment reuses the current FastAPI web app
- Pushes to `main` sync to the matching Hugging Face Space through `HF_TOKEN`
- Public demo mode disables server-side persistence by default
- Anonymous visitors cannot save client data to the server or browse other people's saved data
- Localhost previews for your own advisor workflow should normally enable `FAPM_ENABLE_SERVER_STORAGE=1`
- If `qmc` is not installed in the runtime, playbook generation may degrade and skip return fan charts
