# 版本管理

**版本：** 0.1.0  
**状态：** draft

English version: [07-versioning.md](07-versioning.md)

---

## 1. 概述

本章定义：

- 方法论手册的版本约定
- 应用行为的版本约定
- 剧本合同层的版本约定

版本管理很重要，因为方法论会持续演进。
系统需要区分：

- 措辞修正
- 向后兼容的新规则
- 真正的合同变化

## 2. 版本风格

项目采用类 SemVer 的方式：

```text
MAJOR.MINOR.PATCH
```

含义：

- `MAJOR`
  - 不兼容的方法论或 schema 变化
- `MINOR`
  - 新增兼容规则、新字段或新结构
- `PATCH`
  - 措辞修订、文档清理、小幅默认值调整、局部 bug 级别澄清

## 3. 各级别通常由什么触发

### 3.1 PATCH

适用于：

- 措辞澄清
- 文档清理
- 非结构性渲染修正
- 小幅默认假设调整

### 3.2 MINOR

适用于：

- 新增兼容事件类型
- 新增可选输入字段
- 新增解释性段落
- 在不破坏现有消费者的前提下扩展输出结构

### 3.3 MAJOR

适用于：

- 不兼容 schema 变化
- 核心段落被移除或重定义
- 方法论逻辑变化到足以让旧 profile 的解释失效

## 4. schema version 与 methodology version

在实际运行中，profile 或 playbook 可能会带有版本引用。
其作用是回答：

- 这份档案基于哪一版方法论合同
- 它与当前行为是否仍兼容

## 5. 运行时现实

当前仓库里可能同时存在：

- handbook 版本
- schema 版本
- 静态资源版本

这些不是一回事：

- 方法论合同版本
- 数据兼容版本
- 静态资源缓存失效版本

## 6. 什么时候旧剧本应重算

旧剧本通常在这些情况下值得重算：

- 方法论出现重大变化
- 某个 MINOR 变化已经明显改变解释含义
- 客户事实本身已经变化

并不是每一次 PATCH 级文案修正都需要重算。

## 7. 文档维护责任

当当前代码行为发生实质变化时，应同步更新对应 handbook 章节，使这三者保持一致：

- 当前输出
- 当前文档
- 当前测试

## 8. 向后兼容原则

能兼容时尽量兼容，尤其是：

- 可选字段
- 解释层文案变化
- 增量型输出段落

但不应为了兼容而让当前方法论表达变得模糊。
