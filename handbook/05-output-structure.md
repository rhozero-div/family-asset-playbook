## 1. Purpose of This Chapter

This chapter describes the **actual client playbook structure generated by the current engine**.

The authoritative sources are:

- `engine.markdown_renderer`
- current CLI and web output
- current tests

If legacy examples or older methodology notes conflict with this chapter, this chapter and current rendering behavior take priority.

## 2. What the Current Playbook Is

The current playbook is not:

- a traditional product recommendation report
- a pure narrative essay
- a thin conceptual memo with almost no execution logic

It is a client-facing planning playbook that serves three roles at once:

1. explain the household’s current situation
2. show whether major milestones are covered
3. turn that conclusion into a bucket-based execution discussion

## 3. Current Reading Order

The current playbook is organized in this order:

1. Executive Summary
2. Client Overview
3. Asset Projection
4. Allocation Execution Plan
5. Insurance Structure Suggestion

That reading order is deliberate:

- conclusion first
- facts next
- projection after that
- execution logic after that
- insurance suggestion last

## 4. Executive Summary

The Executive Summary is the front reading layer.
It currently includes items such as:

- milestone coverage conclusion
- long-term surplus-account return conclusion
- stage reading of the household
- top current priorities
- insurance structure suggestion
- monitoring signals
- recalculation triggers

Its role is to help both advisor and client orient quickly before reading the detailed sections.

## 5. Client Overview

This section provides the factual profile layer, including items such as:

- starting assets
- income and spending snapshot
- retirement-related parameters
- milestone table

Its role is to show the factual assumptions the rest of the playbook is built on.

## 6. Asset Projection

This section explains:

- deterministic milestone coverage
- yearly cash-flow and asset path
- return-based total asset path
- milestone table results

It should help the client answer:

- what is the pressure timeline?
- where does the plan hold?
- where does it break, if it breaks?

## 7. Allocation Execution Plan

This section turns the projection into implementation logic.
It currently includes items such as:

- initial capital allocation
- annual net-surplus routing
- bucket balances
- bucket timeline charts
- bucket funding-source charts

This is where the playbook becomes operational rather than descriptive.

## 8. Charts and Tables

The current playbook uses charts and tables as explanatory devices, not decoration.
They are expected to help the client understand:

- timing
- funding order
- range rather than single-point certainty
- the difference between near-term and long-term money
- insurance gaps and the difference between the two insurance suggestion paths

Within the insurance section specifically:

- the playbook itself remains client-facing
- Plan A and Plan B can open an advisor-side insurance input page
- that page is for incremental additions only
- re-running from that page merges the added insurance back onto the original questionnaire and regenerates the playbook

## 9. Tone Requirements

The playbook should remain:

- readable
- planning-oriented
- client-facing
- non-sales-like
- explicit about boundaries

It should not drift into:

- product recommendation language
- guaranteed-return language
- over-technical quant jargon

## 10. Mandatory Boundaries

The current playbook must retain at least these boundaries:

- it is not investment advice
- it does not promise returns
- it does not replace legal, tax, actuarial, or specialist advice

See [`06-boundaries.md`](06-boundaries.md) for the full boundary contract.

## 11. Relationship to Older Definitions

Older materials may still describe the playbook using different structures, such as:

- older narrative sections
- stronger emphasis on conservative / balanced / aggressive display as the main story
- fewer charts or no bucket breakdowns

The current implementation has evolved beyond that.
All current explanations should now use this chapter as the operative output definition.
