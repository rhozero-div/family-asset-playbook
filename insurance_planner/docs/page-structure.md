# 页面与输出结构说明

这份文档说明 `insurance_planner/templates/form.html`、`insurance_planner/templates/report.html` 和 `insurance_planner/templates/apply.html` 当前在页面上如何组织内容。

## 1. 输入页

输入页分左右两栏：

- 左侧
  - YAML 输入
  - 参数输入
  - 提交按钮
- 右侧
  - 这版会做什么
  - 快捷入口
  - 当前假设

### 1.1 当前可调参数

当前页面允许顾问或客户直接调的参数包括：

- 总年保费上限
- 自动预算比例
- 高责任家庭定寿目标倍数
- 低责任家庭定寿目标倍数
- 重疾收入倍数
- 重疾支出年数
- 未成年人重疾目标
- 老人重疾目标
- 是否考虑高端医疗 / 特需升级

这部分的目标是把“方法论里的可调目标参数”显式化，而不是把规则藏在代码里。

## 2. 报告页总体顺序

当前报告页顺序是：

1. 家庭快照
2. 规划逻辑
3. 客户目标参数
4. 识别到的保障缺口
5. 方案 A
6. 方案 B
7. 当前假设与边界

这仍然是一个保留中的独立报告页，主要用于调试和观察保险建议算法。  
主产品默认不会先把用户带到这里，而是先走问卷生成最终剧本。

## 3. 保障缺口区块

这一块先放一张全成员的缺口表，然后按成员逐个展开。

每个成员当前展示：

- 医疗报销状态
- 保障缺口对比图
- 逐人解释文字

这样安排的目的，是让顾问和客户先看“这个人缺什么”，再看“为什么缺、为什么要先补这类保障”。

## 4. 方案 A / 方案 B 区块

方案 A 和方案 B 的结构保持一致：

1. 先给方案概览
2. 再给整张方案分配表
3. 表后按成员分别展示图表和解释

每个成员当前展示：

- 当前 vs 方案后 的医疗状态
- 保额前后对比图
- 逐人解释文字

## 5. 回填页

`apply.html` 是当前更贴近主链路的保险页面。

它的结构是：

1. 顶部说明
2. 方案 A 区块
3. 方案 B 区块

每个方案区块内部：

- 按成员分卡片
- 每个成员一张表
- 每个险种一行
- 只填写“新增”配置，不展示原保单替换逻辑

表里当前录入：

- 基础医疗报销
- 定期寿险
- 重疾险
- 高端医疗
- 每项对应的新增年保费
- 每项对应的剩余缴费年数

这页提交后，会把新增字段叠加回原问卷 YAML，再重新生成剧本。

## 6. 为什么不再单独放“读图提示”

当前版本已经去掉独立的“读图提示”区块。

原因是：

- 这类提示会打断阅读
- 图表本身已经跟在对应表格和成员解释后面
- 如果还额外放一块总提示，信息会重复，而且更像项目说明，不像客户阅读材料

## 7. 逐人解释的文风目标

逐人解释的目标不是：

- 简单重复表格里的数值
- 简单重复“有缺口”“建议加保”

而是：

- 解释这个人在家庭结构里的位置
- 解释为什么优先级高或低
- 解释为什么这次补得厚或补得薄
- 解释为什么某些保障排在后面

当前实现仍然是规则生成，但页面设计已经为未来更自然的解释层留好了位置。
