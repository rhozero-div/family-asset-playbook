# 保险配置建议模块

这是 `family-asset-playbook` 里的保险建议计算模块。
当前主产品链路里，它的结果会并入最终剧本末尾，而不是要求用户单独进入一页。

它的定位不是卖保险，也不是精算报价，而是回答这几个问题：

- 这个家庭当前最需要先保什么
- 在现有收入、资产、负债和未来重大支出压力下，保费预算大概应该压在什么范围
- 如果预算有限，应该优先把哪一层保障做厚
- 如果希望先把结构搭起来，应该怎么分散配置

## 这部分不做什么

- 不做具体保险产品推荐
- 不做保险公司比较
- 不做健康告知或核保结论
- 不做真实保费试算
- 不替代持牌顾问或经纪人的正式投保建议

当前的保费与保额关系仍然采用校准过的 `mock` 费率曲线，作用是帮助比较优先级和预算压力，而不是给出可直接购买的报价。

## 当前主用法

当前主用法是：

1. 通过主问卷生成统一 YAML
2. 直接生成最终剧本
3. 在剧本末尾阅读保险配置建议

仓库里仍保留独立保险页路由，但它不再是当前默认工作流的一部分。

## 当前并入剧本的输出结构

当前并入剧本的保险部分由这几块组成：

1. 家庭快照
2. 识别到的保障缺口
3. 方案 A：优先补足核心保障
4. 方案 B：尽量都配一些，但单项保额更克制
5. 逐人解释与图表

其中“识别到的保障缺口”、方案 A、方案 B 三块，都会按成员分别展示：

- 表格
- 该成员的图表
- 该成员的解释文字

## 代码结构

- `logic.py`
  - 保险需求识别、预算口径、两套方案分配、成员解释文字
- `routes.py`
  - 页面路由、表单参数读取、报告渲染
- `templates/form.html`
  - 输入页模板
- `templates/report.html`
  - 报告页模板
- `tests/test_logic.py`
  - 逻辑测试
- `tests/test_routes.py`
  - 页面路由测试

## 当前实现原则

- 复用主项目已经生成的问卷 YAML，不另造一套输入格式
- 优先做“结构正确”和“逻辑通用”，再考虑更细的市场化保费映射
- 解说文字优先采用规则生成，保证稳定和可控
- 如果未来要升级为更像顾问口吻的解释层，可以在此基础上再接 LLM

## 当前角色规则

当前版本先看家庭责任，再看成员角色，不靠 LLM 判断核心优先级。

- 责任型家庭
  - 有未成年子女、受赡养老人，或家庭有明显负债责任
  - 对主要/次要收入者，通常按 `医疗 -> 定寿 -> 重疾 -> 医疗升级`
- 大众基础配置
  - 没有明确受养责任，或成员更接近独立承担自己的现金流
  - 通常按 `医疗 -> 重疾 -> 定寿 -> 医疗升级`
- 未成年人
  - 重点看医疗和重疾，不作为定寿主建议对象
- 受赡养老人
  - 重点看医疗，重疾只做克制补充，一般不主动建议定寿

对于成年受抚养成员，系统采用一条简单的经济独立判断线：

- 如果预计年收入已经达到 `个人年常规支出 × 1.2`
  - 就按“基本经济独立成年人”处理
- 否则
  - 仍按“家庭支持向独立过渡的成员”处理
